# Yo'nalishlar (Route Configs) — Settings Tab

Predetermined trip/reys config tab under **Sozlamalar**. Frontend-only for now — no API, no backend wiring. Data persists in `localStorage` via zustand.

## What it does

A table of predefined "yo'nalish" records. Each row has:

| Field | Type | Source |
|---|---|---|
| Yuk egasi | combobox | `CLIENT_OPTIONS` (hardcoded) |
| Yuklash manzili | combobox | `LOCATION_OPTIONS` (hardcoded) |
| Yuk tushirish manzili | combobox | `LOCATION_OPTIONS` (hardcoded) |
| Yuk turi | combobox | `CARGO_TYPE_OPTIONS` (hardcoded) |
| To'lov turi | combobox | `PAYMENT_TYPE_OPTIONS` (hardcoded) |
| Valyuta | combobox | `CURRENCY_OPTIONS` (hardcoded) |
| Summa | number input | free text |

Add / edit / delete all work locally. Search is client-side.

## How it's wired

- **Route**: `/route-configs` → `createLazyFileRoute('/_main/_settings/route-configs/')`
- **Menu**: appended to the Sozlamalar items array in `usePaths.tsx`
- **Store**:
  - `useRouteConfigsStore` — `{ items, add, update, remove }`
  - Persisted under localStorage key `route-configs-store`
  - Row id generated via `crypto.randomUUID()` (fallback: timestamp+random)
- **Editing state**: reuses the project-wide `useGlobalStore` pattern with key `ROUTE_CONFIG_KEY = "route-config-item"` so the existing `TableHeader` component's "Qo'shish" button works unchanged (it calls `clearKey(storeKey)` before opening the `create` modal).
- **Create/Edit modal**: `modalKey="create"` (same convention as other settings pages).
- **Delete modal**: `modalKey="delete"` but the confirmation is a local inline `Modal` — NOT the shared `DeleteModal` component, because that one calls `useDelete` (API). Inline version just calls `remove(id)` from the store.
- **Search**: `TableHeader` writes to `route_configs_search` URL param; page reads via `useSearch({ strict: false })` and filters the enriched rows in-memory over client/loading/unloading/cargo_type/payment_type name fields.

## Row rendering

`items` are stored with numeric IDs. The page `enriches` them by looking up names through `findName(...)` against the hardcoded options, producing `RouteConfigRow` with `*_name` strings. Columns just read those strings. `amount` formatted with `fr-FR` locale (space thousands separator).

## When hooking up the real API

When backend is ready, swap these pieces:
1. **Data source**: replace `useRouteConfigsStore` calls in `index.tsx` with `useGet(ROUTE_CONFIGS_ENDPOINT, ...)` and remove the enrichment step (backend should return `*_name` fields, or adapt the columns).
2. **Dropdowns**: replace the hardcoded arrays in `add-route.tsx` with `useGet` on `SETTINGS_SELECTABLE_CLIENT` / `SETTINGS_SELECTABLE_DISTRICT` / `SETTINGS_SELECTABLE_CARGO_TYPE` / `SETTINGS_SELECTABLE_PAYMENT_TYPE` (these already exist in `api-endpoints.ts` — see how `create-reys.tsx` uses them).
3. **Mutations**: replace `add` / `update` in `add-route.tsx` with `usePost` / `usePatch`, and `remove` in `index.tsx` with `useDelete` (or just use the shared `DeleteModal`).
4. **Delete modal**: drop the inline version in `index.tsx` and use the shared `DeleteModal path={ENDPOINT} id={item?.id} />`.
5. Delete `options.ts` and `route-configs-store.ts` once migration is complete.

## What was intentionally NOT touched

- `src/pages/home/managers/manager-reys/create-reys.tsx` — the "reys qo'shish" modal. The user explicitly said to leave it alone. Eventually these route-configs should feed into it as a "preset" picker, but that's a follow-up task.

## Known caveats

- `routeTree.gen.ts` is auto-generated by the TanStack router Vite plugin on dev start. It got regenerated automatically when the new `index.lazy.tsx` was created; don't hand-edit it.
- The `TableHeader` component's `fileName` prop is used for Excel export which is commented out in the component, so the value ("Yo'nalishlar") is cosmetic.
- Delete modal uses the shared `"delete"` modal key — safe because only one route is mounted at a time, but be aware if that changes.
